# Plugdo｜AI自動Web診断ツール 追補仕様書（v2：7軸化＋厳格化）

> 本書は既存の「AI自動Web診断ツール 仕様書」への**差分・追補**です。
> 目的：診断が甘く高得点に寄る問題を解消し、(1) 観点を5軸→7軸に拡張、(2) 内部採点項目を拡充、(3) スコアリングとClaudeプロンプトを「厳しめ・正直」に校正する。
> 差し替え対象：既存仕様の §3（診断軸）／§4（スコアリング）／§8（Claudeプロンプト）／§5（結果画面）の一部。

> 実装メモ（2026-06-26）：⑦表示の速さは PageSpeed Insights ではなく Worker側ヒューリスティック（HTML総量・画像数・WebP・遅延読込・レンダリング阻害）で実装。理由＝PSIは1回15〜40秒かかり「30秒診断」のUXを壊す＋Google APIキー追加が必要なため。PSIは後日オプションで追加可能。

## 0. 設計原則（厳格化の指針）

厳しくする方向は2種類あり、**正しい厳しさだけを採用する**。

- ✅ 正しい厳しさ：**採点基準を引き上げる**（測る項目を増やす／合格ラインを上げる／段階評価・減点方式）。嘘をつかずスコアが下がる。
- ❌ 危ない厳しさ：実態に関係なくスコアを低く出す。良いサイトで回すと「診断がおかしい」となり、本体の無料診断の信頼まで壊す。**禁止。**

補助原則：**スコアは正直に、でも予約理由は常に残す。** 高得点サイトにも「次のレベルの改善案」を必ず提示する（85点→95点にする2点、など）。低スコアで釣る必要はない。

## 1. 見せる軸：5軸 → 7軸

表に見せる軸は**7つが上限**（ITリテラシーが高くない経営者向け）。既存5軸に2軸を追加する。

| | 軸 | 経営者向けの問い | 配点 |
|---|---|---|---|
| ① | 第一印象・信頼感 | 3秒で「ちゃんとした会社」に見えるか | 15% |
| ② | 伝わりやすさ | 何の会社で、なぜここに頼むか分かるか | 20% |
| ③ | 問い合わせのしやすさ | 迷わず問い合わせ・電話まで行けるか | 20% |
| ④ | スマホ対応 | スマホで見て使いやすいか | 12% |
| ⑤ | 見つけてもらえるか | 検索・AIに正しく拾われるか | 13% |
| **⑥** | **信用の証拠**（新規） | 実績・お客様の声・資格などで安心させているか | **10%** |
| **⑦** | **表示の速さ**（新規） | ページがすぐ表示され、離脱されないか | **10%** |
| | | **合計** | **100%** |

配点の考え方：変換の主レバーである②伝わりやすさ・③問い合わせを最重視。新規⑥⑦は「中小サイトの大半が弱い」領域なので、実態に即してスコアが下がる。

## 2. 新規軸の判定ロジック

### ⑥ 信用の証拠（10%）

**決定的チェック（コードで判定）**
- お客様の声／レビュー欄／事例の有無
- 実績・数値（「◯年」「◯件」「合格率」等）
- 資格・受賞・メディア掲載・所属団体
- 会社概要／代表者情報
- 更新の鮮度（当年・前年の日付）

**Claude定性判定**
- 証拠が「具体的で信じられる」か（実名・顔写真・第三者性、抽象的な自称でないか）

### ⑦ 表示の速さ（10%）

**実装（Worker側ヒューリスティック）**
- HTML総量／画像点数／WebP利用／lazy-load／レンダリング阻害資源（head内の同期script＋stylesheet数）
- 減点方式で段階評価

## 4. スコアリング設計（厳格化版）

### 4-1. 段階評価
各項目を「無=0／あるが不十分=部分点／できている=満点」の3段階＋減点方式。存在チェックを廃止し質まで見る。

### 4-2. スコア分布（校正目標）
- **80〜100**：明確に優れている（少数）
- **50〜65**：平均的な中小企業サイト（大多数）
- **〜40**：放置・課題が多い

### 4-3. 点数帯メッセージ
- 80〜100：土台は優秀。あと数点で“働くサイト”に
- 66〜79：良い部分と伸びしろが混在。優先順位づけの価値大
- 50〜65：平均的。直せば問い合わせが増える余地が大きい
- 〜49：もったいない状態。ただし伸びしろは最大

## 5. Claude APIプロンプト（辛口校正版）

- 平均的SMEが50〜65点になる基準。80点以上は明確に優れている場合のみ。
- 各減点に具体的根拠 evidence を必須（根拠なき減点は禁止）。
- 「存在するか」でなく「質」で見る。専門用語禁止。「もったいない／伸びしろ」表現。
- 評価軸：clarity / impression / credibility。
- 分布アンカー（採点例）をプロンプトに同梱。
- 出力JSON：clarity_score / impression_score / credibility_score / findings[{axis, text, evidence}]。
- findings は問題の指摘まで（直し方は書かない＝ゲート）。
