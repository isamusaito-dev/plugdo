# Plugdo｜AI自動Web診断ツール 仕様書

> 目的：URLを入れるだけの自動診断で気づきを与え、**「無料診断（人による本格診断）」の予約につなげる**
> 対象：中小企業の経営者（ITリテラシーは高くない前提）
> 設計の核：**問題は見せる／直し方はゲートする**。正直なスコアでブランドを守る

---

## 1. 製品コンセプト

| | 内容 |
|---|---|
| 一言で | サイトのURLを入れると、30秒で「問い合わせにつながる状態か」を採点する自動診断 |
| 本当の役割 | リードマグネット。最終ゴールは無料診断の予約（自動診断はその"入口の入口"） |
| 体験のゴール | 「うちのサイト、思ったより損してたかも」と気づかせ、次の一歩を自然に踏ませる |
| 置き場所 | `plugdo.jp/check`（トップのヒーローから大きく導線。サイト内の主要CV装置に） |

### コンバージョン哲学（全画面これで貫く）

1. **問題は具体的に見せる**（自分のサイトの実際の課題を名指し＝信頼と"焦り"）
2. **直し方・優先順位はゲートする**（それは無料診断で人が解説）
3. **次の一歩を極限まで軽く**（無料・30分・契約不要）

---

## 2. ユーザー体験フロー（画面ごと）

### 画面① 入口（URL入力）

- ファーストビューに**入力欄1つ＋大きなボタンだけ**。迷う要素ゼロ
- コピー例：
  - 見出し「そのホームページ、問い合わせにつながっていますか？」
  - サブ「URLを入れるだけ。30秒で、プロの視点であなたのサイトを採点します。」
  - ボタン「無料で診断する」
- 入力補助：`https://` を自動補完。プレースホルダに例示
- 信頼要素を小さく添える：「プロのWebデザイナーが設計した診断ロジック」

### 画面② 診断中（演出＝体感価値）

- 20〜30秒。**チェック項目が順に光る**アニメーション（既存のフローラインFVの美学を流用）
- 表示する項目（専門用語は使わない）：
  - 「第一印象をチェック中…」
  - 「何の会社か伝わるか確認中…」
  - 「問い合わせのしやすさを確認中…」
  - 「スマホ表示をチェック中…」
  - 「検索・AIに見つけてもらえるか確認中…」
- 待ち時間を"丁寧に診てくれている"という価値に変換する

### 画面③ 結果（リビール＝最大の山場）

表示順（上から）：

1. **総合スコア（0〜100）** を大きなゲージで。感情のアンカー
2. **5軸のスコア**（赤／黄／緑で色分け。直感的に「赤がやばい」と分かる）
3. **見つかった気づき 2〜3点**（自分のサイトの実例を、平易な言葉で名指し）
4. **ゲート**：「あと◯個の改善ポイントが見つかっています。具体的な直し方と"何から手をつけるべきか"は、無料診断でプロが解説します」
5. **CTA（主）**：「この結果をもとに、無料で詳しく診断してもらう」
6. **CTA（副）**：「結果をメールで受け取る」（メール取得→ナーチャリング）

### 画面④ 申込（予約フォーム）

- 画面③のURL・スコアを**引き継いで前埋め**（連続体験。入力負荷を減らす）
- 項目：会社名・お名前・メール・（任意）電話・相談内容
- 既存 contactページの診断申込フォームと統合

## 3. 診断の5軸（中小企業の言葉で）

専門用語を一切出さず、経営者が「自分ごと」で分かる軸にする。

| 軸 | 経営者向けの問い | 重み |
|---|---|---|
| ① 第一印象・信頼感 | 3秒で「ちゃんとした会社」に見えるか | 20% |
| ② 伝わりやすさ | 何の会社で、なぜここに頼むべきか分かるか | 25% |
| ③ 問い合わせのしやすさ | 電話・問い合わせまで迷わず行けるか | 25% |
| ④ スマホ対応 | スマホで見て使いやすいか（訪問の6割はスマホ） | 15% |
| ⑤ 見つけてもらえるか | 検索やAIに正しく拾われるか | 15% |

### 各軸の判定ロジック

**① 第一印象・信頼感（20%）**

- HTTPS化されているか（信頼の最低条件）
- ファビコン設定の有無
- デザイン品質（後述のスクリーンショット×Claude判定。または初期はHTML/CSSの手がかりで簡易判定）
- レイアウトの現代性（レスポンシブか、極端に古い構造でないか）

**② 伝わりやすさ（25%）── Claude APIで定性判定**

- ファーストビューで「何の会社か」が分かるか
- 「誰向けか」が分かるか
- 「なぜここを選ぶか（強み・差別化）」が言えているか
- コピーが独りよがりでないか

**③ 問い合わせのしやすさ（25%）**

- ヘッダー/ナビに問い合わせリンクがあるか
- フォームの有無
- 電話番号リンク（`tel:`）の有無・視認性
- ファーストビュー内にCTAがあるか
- CTAがページ内で繰り返されているか

**④ スマホ対応（15%）**

- viewport設定の有無
- レスポンシブ対応（メディアクエリ／固定幅でないか）
- 文字サイズ・タップ領域のヒューリスティック

**⑤ 見つけてもらえるか（15%）**

- タイトルの設定・品質（未設定/重複の検出）
- 検索結果に出る説明文（meta description）の有無
- 見出し（h1）の有無・構造
- 構造化データ（JSON-LD）の有無＝AI検索対応の指標
- 画像の代替テキスト網羅率

---

## 4. スコアリング設計

- 各軸 0〜100点 → 重み付き合計で総合点
- **正直に出す。** 良いサイトは80〜90点が出るよう校正する（全部低得点にしない）
- 点数帯のメッセージ（経営者を責めず、機会として見せる）：
  - 80〜100：「土台は良好。あと一歩で"働くサイト"に」
  - 60〜79：「惜しい。直せば問い合わせが増える余地が大きい」
  - 40〜59：「もったいない状態。優先順位をつけて直す価値あり」
  - 〜39：「機会を逃しています。でも伸びしろは最大」
- **重要**：低スコアでも「ダメ」ではなく「もったいない＝伸びしろ」と表現する

---

## 5. 結果画面のコンバージョン設計（最重要）

| 見せるもの（自動） | ゲートするもの（無料診断へ） |
|---|---|
| 総合スコア | 各課題の具体的な直し方 |
| 5軸の色分け | 改善の優先順位（何から手をつけるか） |
| 気づき2〜3点（名指し） | 月次予算の目安・進め方 |
| 「あと◯個ある」という残数 | サイトに合わせた個別レポート |

### 効かせる心理レバー

- **ギャップ／損失回避**：低スコアが「直したい」を生む
- **具体性＝信頼**：実際の自社の課題を名指しすると一気に信用される
- **残数の提示**：「あと◯個」が次の一歩を引く
- **権威**：「プロのデザイナーが設計した診断」
- **超低コミットの次の一歩**：無料・30分・契約不要

## 6. UI/UXデザイン指針（プロ品質）

**ブランド**

- カラー：ネイビー `#103366` / ピンク `#E43172`
- 見出しフォント：Poppins（SemiBold/Bold。ロゴと統一）
- 本文：可読性の高い和文ゴシック

**レイアウト**

- **モバイルファースト**（対象の訪問の大半がスマホ）
- 大胆な余白・自信のある大きめタイポ
- 結果のゲージを主役のビジュアルに
- 軸スコアは色付きバーで一目で分かるように

**マイクロインタラクション**

- 診断中：項目が順に点灯／プログレス
- スコア：ゲージが0から実数までアニメで上がる
- 結果リビール：上から要素が順に現れる

**やってはいけないこと（低リテラシー対策）**

- 専門用語を出さない（×メタディスクリプション → ○検索結果に出る説明文）
- データを浴びせない（軸は5つまで、気づきは2〜3点）
- 相手を馬鹿にしない（「ダメ」でなく「もったいない」）

## 7. 技術アーキテクチャ（既存スタックに乗せる）

```
[ブラウザ/Astro]
   │ URL送信
   ▼
[Cloudflare Worker / Functions]  ← CORS回避のためサーバー側で対象サイトを取得
   │ ① 対象サイトのHTMLを fetch
   │ ② 決定的チェック（軸③④⑤と①の一部）をコードで判定
   │ ③ スクリーンショット取得（任意：Cloudflare Browser Rendering 等）
   │ ④ Claude API 呼び出し（軸①②の定性判定＋平易な気づき文の生成）
   ▼
[結果JSON] → フロントで美しくレンダリング
```

- ホスティング：Cloudflare Pages（既存）
- 取得：Worker側で `fetch`（クライアントのCORS問題を回避）
- スクショ：Cloudflare Browser Rendering API（第一印象をClaudeに"見せる"なら。MVPでは省略可）
- 定性判定・文章生成：Claude API（`claude-sonnet` 系）
- 申込：既存 Formspree／contactフォームに連携

## 8. Claude API プロンプト設計（定性判定のコア）

抽出した本文・見出し・（あれば）スクショを渡し、**JSONのみ**を返させる。

**システムプロンプト（例）**

```
あなたは中小企業のWebサイトを診断するプロのUI/UXデザイナーです。
与えられたサイト情報をもとに、以下を中小企業の経営者にも分かる平易な日本語で評価してください。
専門用語は禁止。相手を責めず「もったいない／伸びしろ」という前向きな表現で。

評価軸：
- clarity（伝わりやすさ）: 何の会社か・誰向けか・なぜ選ぶかが伝わるか
- impression（第一印象・信頼感）: 3秒で信頼できる会社に見えるか

出力は次のJSONのみ。前置き・マークダウン・コードフェンス禁止：
{
  "clarity_score": 0-100,
  "impression_score": 0-100,
  "findings": [
    {"axis": "...", "text": "経営者向けの平易な気づき（直し方は書かない・問題の指摘まで）"}
  ]
}
findings は必ず「問題の指摘」までに留め、「どう直すか」は書かないこと。
```

**ユーザーメッセージ**：抽出した `title` / FV本文 / 見出し構造 / （任意で）スクショ画像を渡す。

**実装注意**

- 返り値は コードフェンス除去 → `JSON.parse`、失敗時フォールバック
- `findings` の "直し方は書かない" 制約が、ゲート設計の生命線

---

## 9. エッジケース・信頼性

- 無効URL／取得失敗：「サイトを読み込めませんでした。URLをご確認ください」＋再入力
- 巨大サイト：トップページのみ対象と明示
- 非日本語サイト：対応範囲を明示
- 同一URL連打：簡易キャッシュ
- **スコアは正直に**。良いサイトは高得点。これがブランドと本診断の信頼を守る

---

## 10. 計測（最適化のため）

- URL入力数 → 診断完了数 → 結果表示数 → 申込クリック → 予約完了
- どのスコア帯が最も予約に転換するか
- 副CTA（メール受取）の取得数
- これらで結果画面のコピー・CTAを継続改善

---

## 11. フェーズ分け

**MVP（まず7月に出す）**

- URL入力 → Worker取得 → 決定的チェック（③④⑤＋①一部）＋ Claude定性判定（①②）→ スコア＋気づき → 予約CTA
- スクショは省略可（HTML/テキストベースで判定）

**強化（8月以降）**

- スクリーンショット×Claude visionで第一印象の精度UP
- 結果のメール送付（PDF）→ ナーチャリング
- 業種別の比較ベンチマーク（「同業の平均は◯点」）
- 結果のSNSシェア導線（拡散＝集客の自動化）

---

## 12. 最初の一歩

MVPの肝は **Worker でのサイト取得 ＋ Claude定性判定 ＋ 結果画面のゲート設計**。
ここが動けば、メール営業のウォームアップ問題に縛られない、24時間無人で回るリード装置になる。
